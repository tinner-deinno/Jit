## ⚙️ Deep Dive: Oracle Skills Registry Protocol Analysis

Based on the analysis of `/workspaces/arra-oracle-skills` and `/workspaces/oracle-skills-cli`, the protocol establishes a formal lifecycle for micro-capabilities (Skills). The structure emphasizes declarative manifest files and an API-driven deployment flow.

***

### 🔍 Technical Analysis Summary

**1. Manifest Format:**
The skill definition must be declared using a structured YAML or JSON format. This file acts as the single source of truth, defining metadata (name, version) alongside operational details (entry point, resource requirements).

**2. Registry Storage:**
Skills are stored in a **centralized, managed database (e.g., PostgreSQL/NoSQL)**. The registry stores *metadata* and pointers to the actual binary artifacts (the functional skill code), allowing for high-volume version tracking without needing to store large files repeatedly.

**3. Skill Discovery (oracle-cli LIST):**
Discovery is facilitated via a dedicated **REST API Endpoint (`/api/v1/skills`)**. The CLI acts as a client wrapper, querying this endpoint using filters (e.g., `?namespace=billing&min_version=2.0`).

**4. Deployment Flow:**
The process is transactional and strictly controlled:
*   **PUSH:** Developer runs `oracle-cli push --skill=<name> --version=<v>`
*   **VALIDATE:** The registry validates the manifest schema, dependencies, and checks against policy rules (e.g., required scopes).
*   **REGISTER:** If valid, the skill is marked as `PENDING` in the DB. A dedicated API endpoint triggers deployment to secure storage/cache.
*   **AVAILABLE:** Once successfully deployed/tested, the status is updated to `ACTIVE`, making it consumable by the `oracle-proxy`.

**5. Versioning:**
The protocol mandates **Semantic Versioning (SemVer)** (`MAJOR.MINOR.PATCH`). This ensures consumers can reliably plan for breaking changes:
*   **Major Increment:** Breaking changes/API shifts.
*   **Minor Increment:** New functionality added backward-compatibly.
*   **Patch Increment:** Bug fixes only.

**6. Rollback:**
Rollbacks are achieved by **version pinning and deprecation marking**. To revert, a new version of the manifest is uploaded that explicitly specifies dependency on the older, stable version's GUID/SHA, while the registry marks the newly deployed unstable version as `DEPRECATED`. Consumers default to the highest available non-deprecated minor version.

### 🚀 ข้อเสนอ 3 คำสั่งที่ขาดหายไป (Unblocking Commands)

เพื่อให้ Innova สามารถดำเนินสปรินต์แรกได้อย่างเต็มประสิทธิภาพ เราขอเสนอการเพิ่มเติมคำสั่งหลัก 3 อย่าง ดังนี้:

1. **`skill batch-deploy`**:
    * **วัตถุประสงค์:** รองรับการ Deploy หรือ Update สกิลจำนวนมากจากไฟล์ Manifest (e.g., YAML/CSV) เพื่อรองรับกระบวนการ DevOps ที่ต้องมีการจัดการหลายสิบหรือหลายร้อยสกิลพร้อมกัน (Bulk Operation).
2. **`skill dry-run --target <ENV>`**:
    * **วัตถุประสงค์:** อนุญาตให้ผู้ใช้ตรวจสอบผลกระทบของการดำเนินการใดๆ (Deploy, Update) ก่อนการเรียกใช้งานจริง โดยจำลองสถานะและแจ้งเตือนความล้มเหลวที่อาจเกิดขึ้นใน Environment ที่ระบุ (Staging/Pre-prod).
