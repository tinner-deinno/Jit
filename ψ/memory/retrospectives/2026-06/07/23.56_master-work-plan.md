## Lessons Learned

### 1. Fan-Out Planning Requires Normalized Input Data
When analyzing multiple tickets across categories, first normalize all tickets into a single JSON schema (id, title, priority, owner, effort, type, dependencies). Then categorize and sort. This avoids manual reconciliation when merging subagent results. **Generalizable**: Any multi-agent analysis task benefits from a shared schema.

### 2. Critical Path Emerges from Dependencies, Not Priority
The P0 tickets aren't necessarily the critical path. JIT-001 (TTL, P0) blocks JIT-002 (Idempotency, P0), which blocks JIT-003 (Retry, P1), which blocks JIT-004 (DLQ, P1). A P1 ticket on the critical path is more urgent than a standalone P0. **Generalizable**: Always map dependencies before prioritizing sprint order.

### 3. Documentation Should Precede Implementation by 1 Week
vaja's documentation stream has a dependency graph where JIT-019 (body-map.md) unlocks all other docs. By scheduling docs 1 week ahead of features, specs are ready when engineers start. This prevents "code first, docs later" drift. **Generalizable**: Docs lead, not lag, in multi-agent systems where specs guide implementation.

### 4. Workload Imbalance is Invisible Without By-Agent Aggregation
innova has 37 hours of tickets, while neta, rupa, and karn have zero. Without aggregating by owner, this imbalance was hidden. Now innova can delegate or reschedule before sprint start. **Generalizable**: Always show workload distribution across team members in sprint plans.

### 5. Security Tickets Must Be Batched by Component, Not Priority
JIT-006 (hardcoded token) and JIT-022 (token logging) are both pada's work and both involve credential handling. Doing them together saves context-switching even if one is P0 and one is P1. **Generalizable**: Batch by cognitive similarity, not just priority.
