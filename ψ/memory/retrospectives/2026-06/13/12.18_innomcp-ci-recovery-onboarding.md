## Lessons Learned

1. **When fixing a test failure: read the entire test file before fixing source code.** Tests often duplicate source functions for isolation — fixing only the source while the test has its own copy does nothing. Check for comments like "must stay in sync with X."

2. **When debugging a chain of CI failures: map the full dependency chain before fixing anything.** CI failures are often layered (pnpm version → lockfile format → Node compatibility → test suite). Fixing the first visible layer will reveal the next. It's faster to enumerate all likely layers upfront than to fix iteratively.

3. **Rebase on 20+ commits with active conflicts → create a fresh branch instead.** `git checkout origin/main -b fix/name && git checkout <source-branch> -- <specific-files>` is faster and safer than rebasing a long branch with conflicts.

4. **Goal conditions need concrete verification criteria at goal-setting time, not at evaluation time.** ">100 CC sub-agents" is ambiguous. Before accepting a goal, ask: how will I know when this specific condition is met?

5. **Avoidance pattern: choosing clear work over ambiguous work when both matter.** Tactical clarity (CI green) can crowd out strategic goals (architecture, monitoring). Notice this and explicitly timebox the strategic work first.
