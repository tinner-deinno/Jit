## Architecture Decisions

### 1. Heartbeat-Triggered vs Daemon
**Decision**: Use existing heartbeat pulses instead of separate daemon
**Rationale**: 
- Heartbeat already runs every 15 min (adaptive: 5-60 min depending on activity)
- No new long-running process to manage
- Lighter resource footprint (~6 sec/hour overhead)
- Integrates naturally with Jit's pulse system
- Decision: ✅ Correct (lightweight, maintainable)

### 2. Registry vs Direct Queries
**Decision**: Pre-computed registry (trace-registry.json) + lightweight query script
**Rationale**:
- Full `git log` parse is expensive (~10 sec every 90 min)
- Registry allows sub-100ms queries for agents
- Tradeoff: Registry is 90 min behind, but queries are instant
- Use case: Agents need sub-second response (not real-time analysis)
- Decision: ✅ Correct (agents need speed)

### 3. Directory Structure (Date-Based)
**Decision**: `ψ/memory/traces/YYYY-MM-DD/` layout + daily `summary.md` + hourly `index.md`
**Rationale**:
- Time-organized for easy historical navigation
- Summary per day captures daily patterns
- Index per hour captures minute-level activity
- Friction score computed fresh daily
- Decision: ✅ Correct (aligns with human timescale)

### 4. Modular Scripts vs Monolithic
**Decision**: 5 separate scripts (startup, commits, heartbeat, hooks, query) vs one big script
**Rationale**:
- Each script has single responsibility (Unix philosophy)
- Easier to test, debug, update independently
- Reusable components (e.g., trace-query.sh called from multiple contexts)
- Easier to add queries without touching core analysis
- Decision: ✅ Correct (maintainability, extensibility)

### 5. Agent Access Pattern
**Decision**: Query via `bash limbs/trace-query.sh [command]` instead of direct JSON read
**Rationale**:
- Provides stable interface (hides JSON schema changes)
- Human-readable output (tables, markdown)
- Can add business logic later (filtering, aggregation) without changing agents
- Shell script is more portable than requiring JSON parser
- Decision: ✅ Correct (abstraction, stability)

## Lessons Learned

### 1. Modular architecture saves debugging time
When I broke trace system into 5 independent scripts, each tested separately before integration, I caught issues early. If I'd built one monolithic script, the first integration error would have been much harder to debug.

**Takeaway**: When building systems with multiple components, test each component in isolation first. Add integration friction later (it's better to understand each piece deeply than to debug the whole system at once).

### 2. Assumptions about existing infrastructure need verification
I designed around assumptions about heartbeat.sh's structure without verifying. It worked out, but it was luck. The right pattern: read the file first, *then* design.

**Takeaway**: For "integrate with existing system" tasks, invest 2-5 minutes upfront to read the target files. Prevents 30+ minutes of rework during integration.

### 3. Documentation is force-multiplier for integration
Because I created detailed TRACE_INTEGRATION.md with explicit line numbers and context, the integration phase will go smoothly. If I'd just handed off a script without docs, it would be much harder for the next agent.

**Takeaway**: Comprehensive documentation isn't overhead—it's the difference between a 15-minute integration and a 3-hour debugging session.

### 4. Tight feedback loops catch problems early
Testing `trace-startup.sh` immediately after writing it caught issues I would have missed if I'd written all 5 scripts first and tested them together. Early testing = high confidence.

**Takeaway**: Don't batch-write all code then test. Write + test each component in tight loops. Adds 5-10 min total but saves 30+ min in debugging.

### 5. Defensive design reduces coupling risk
I designed the heartbeat integration to be a loosely-coupled "hook," which means even if the heartbeat.sh structure is different than I assumed, the trace system can still be adapted. This is the Unix philosophy in practice: small, focused tools that can be combined in multiple ways.

**Takeaway**: Over-coupled systems are fragile. Over-engineered systems are slow. Aim for the sweet spot: simple, focused components with clear boundaries.
