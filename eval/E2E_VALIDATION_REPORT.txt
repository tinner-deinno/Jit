INTEGRATION FINDINGS (3 DEFECTS)
================================================================================

FINDING #1: routingKey() API misuse in 007b test
──────────────────────────────────────────────────────────────────────────────
Severity: CRITICAL (test harness bug, not implementation)
Status: DOCUMENTED (implementation correct)

Issue:
  The routingKey() function requires a message array: routingKey([{role, content}])
  The 007b test calls: routingKey(prompt) where prompt is a bare string

  Correct signature:
    routingKey(messages, options) where messages is Array<{role, content}>

  007b test sections A/B/C call:
    const k = routingKey(p) where p is "จิตคืออะไร" (string)
    Result: returns "" (empty string) because !Array.isArray(messages)

Impact:
  - Section A/B pass vacuously (all keys empty → trivially "stable")
  - Section C: pickBackendByKey("", [...]) → hash % order always returns BACKEND_ORDER[0]
  - Sections A/B do NOT validate routing key stability; they validate nothing

Fix (for 007b test):
  Change: const k = routingKey(p)
  To:     const k = routingKey([{role: 'user', content: p}])

Status in E2E test: IDENTIFIED and DOCUMENTED


FINDING #2: pickBackendByKey signature mismatch
──────────────────────────────────────────────────────────────────────────────
Severity: CRITICAL (test misuse, not implementation)
Status: DOCUMENTED (implementation correct)

Issue:
  The 007b test calls: pickBackendByKey(key, backends, preferBackend)
  The function signature: pickBackendByKey(key, backends) — 2 parameters only

  Implementation (model-router.js:1223):
    function pickBackendByKey(key, backends) { ... }

  007b test section C (line 112):
    const chosen = pickBackendByKey(key, ALL_BACKENDS, be);
    // Third argument 'be' (preferBackend) is silently ignored

Impact:
  - Section C of 007b fails 8/9 tests (only 'ollama_mdes' always returned)
  - preferBackend is NOT implemented in pickBackendByKey
  - Callers expecting preferBackend to work will silently degrade

Notes:
  - proxy-thai.js also calls with 3 args: pickBackendByKey(keyObj.key, router.status().order, preferBackend)
  - The proxy's preferBackend is also silently ignored
  - Both locations silently accept and ignore the 3rd argument

Fix (architectural):
  Either:
    a) Add preferBackend param to pickBackendByKey signature
    b) Remove preferBackend from all callers
  Recommend: (a) — preferBackend should be supported for routing hints

Status in E2E test: IDENTIFIED and DOCUMENTED
Status of proxy: FUNCTIONAL but preferBackend not honored


FINDING #3: status().backends missing innova_bot
──────────────────────────────────────────────────────────────────────────────
Severity: MEDIUM (reporting gap, not routing gap)
Status: DOCUMENTED

Issue:
  BACKEND_ORDER includes innova_bot (position 7):
    ['ollama_mdes', 'thaillm', 'commandcode', 'ollama_cloud', 'copilot', 'openai', 'innova_bot', 'ollama_local', 'openclaude']

  status().backends does NOT include innova_bot entry:
    ['copilot', 'openai', 'ollama_mdes', 'thaillm', 'commandcode', 'ollama_local', 'ollama_cloud', 'ollama', 'openclaude']

Impact:
  - innova_bot is routable (in BACKEND_ORDER)
  - innova_bot is not reported (missing from status.backends)
  - Monitoring/health checks cannot see innova_bot status
  - Fallback chains may route to innova_bot even if unavailable

Fix:
  Add innova_bot entry to status().backends dictionary (model-router.js:1107-1129)

Status in E2E test: IDENTIFIED and DOCUMENTED
